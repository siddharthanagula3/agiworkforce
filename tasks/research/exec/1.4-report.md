## 1. What shipped

### 1.1 Canonical TypeScript implementation (`@agiworkforce/runtime`)

Two new files in `packages/runtime/src/queue/`:

- `types.ts` — public type surface: `QueuePriority` (`now`/`next`/`later`),
  `QueuedCommand`, `MessageQueue`, `PastedContent`, `ContentBlock`,
  `QueueStorageAdapter`, `PopAllEditableResult`, plus the
  `LANE_CAP = 100` and `PRIORITY_ORDER` constants and two custom error
  classes (`QueueFullError`, `QueueDequeueRaceError`).
- `messageQueueManager.ts` — `createMessageQueue(options)` factory plus two
  storage-adapter helpers (`createWebStorageAdapter` for `Storage`-shaped
  backends, `createKvStorageAdapter` for synchronous KV stores like MMKV /
  `chrome.storage.local` wrappers).

The queue is a **per-surface factory**, not a module-level singleton — each
call to `createMessageQueue()` returns an isolated instance, satisfying
the "Cross-surface state: do NOT share queue state across surfaces" edge
case from §1.4 of the execution plan. State is stored inside a
`createStore<readonly QueuedCommand[]>` (Foundation 1.3, branch
`task-1.3-createstore-onchange`) so the queue inherits the
`Object.is`-short-circuit + `useSyncExternalStore`-friendly snapshot
contract for free.

A `queue/index.ts` barrel re-exports the public API; `runtime/src/index.ts`
re-exports the queue types from the package root, so consumers import
`createMessageQueue`, `QueueFullError`, `MessageQueue`, etc. directly from
`@agiworkforce/runtime`.

## 2. Edge cases (one-by-one)

All seven edge cases from §1.4 of the plan are handled:

1. **`popAllEditable` reconstruction** — preserves `PastedContent.id` in
   insertion order (test: "preserves PastedContent IDs in insertion order"
   asserts `[1, 2, 5]` after enqueuing two commands with
   `pastedContents` `{1, 2}` and `{5}`). For `ContentBlock[]`-embedded
   images the queue allocates sequential IDs starting at `Date.now()`, so
   restored images don't collide with future `Date.now()`-keyed allocations
   in `imageStore`.

2. **Queue overflow** — `LANE_CAP = 100` exported from `types.ts`. Each
   lane (`now`, `next`, `later`) is capped independently. Over-cap enqueue
   throws `QueueFullError` with `lane` and `cap` fields so callers can
   tailor the user-visible message ("queue lane 'next' is full" surfaces
   correctly across all four UI surfaces). Test
   "rejects with QueueFullError when a lane is at cap" enforces the bound,
   "different lanes have independent caps" proves no cross-contamination,
   "default cap is 100 per lane" asserts the constant.

3. **Cross-surface state** — the queue is a factory, not a singleton, so
   multiple instances coexist without contamination. Test "two queues are
   completely independent" enqueues into each separately and verifies size
   stays disjoint.

4. **Persistence** — `next` and `later` lanes serialize through
   `QueueStorageAdapter.write` after every mutation that changes a
   persisted lane. The `now` lane is filtered out by spec (urgent
   messages don't survive a process death). On instantiation, the queue
   reads `storage.read()` and restores all non-`now` commands. Tests
   "persists next and later lanes to storage on mutation",
   "does NOT persist now lane", and "restores next + later lanes on next
   instantiation" cover the round-trip.

5. **Atomic compare-and-swap dequeue** — `dequeueIf(expectedId)` succeeds
   only when the highest-priority head's `id` matches `expectedId`;
   otherwise it throws `QueueDequeueRaceError`. Useful when a consumer
   peeks first (to inspect the command), defers a side-effect, then wants
   to commit ownership without risking a parallel `dequeue()` having moved
   the head. Tests cover (a) the success path, (b) loss-of-head when
   another consumer raced ahead, (c) loss-of-head when a higher-priority
   command was enqueued between peek and commit.

6. **Cancellation via `AbortSignal`** — `enqueue(cmd, { signal })` wires an
   `'abort'` listener that removes the command from the queue. Calling
   `enqueue` with an already-aborted signal short-circuits and does not
   add the command. After successful `dequeue`, the listener is detached
   (`detachAbort(commandId)`) so a later `signal.abort()` is a no-op and
   doesn't leak. All three paths covered by tests in the "AbortSignal
   cancellation" suite.

7. **Backpressure** — by design, the queue is _additive_ on top of the
   existing send pipeline: it does not introduce new asynchrony. Each
   surface's send call enqueues then drains immediately, so the queue
   acts as a structured admission gate. When the LLM is rate-limited and
   prior commands are still in flight (the queued state from prior
   non-drained sends), new enqueues land in the lane and absorb up to
   `LANE_CAP`; past that, the structured `QueueFullError` is the
   backpressure signal. The current rollout keeps the dequeue immediate
   to match existing behavior, but follow-up work (planned Sprint B) can
   defer dequeue when the streaming client signals throttling.

## 3. Architectural decisions

Three decisions worth flagging for review:

1. **Queue is per-surface, factory-built — not a module-level singleton.**
   The reference implementation in
   `~/Desktop/reference/src/utils/messageQueueManager.ts` is a
   module-level singleton. That works for Claude Code (single-process
   CLI) but does not match this codebase's six surfaces, where a desktop
   instance, a web instance, etc. need isolated queues. The factory
   pattern (`createMessageQueue(options)`) plus the
   `getSendQueue(surfaceId)` cache-by-id wrapper preserves the singleton
   ergonomics for callers (one queue per surface) while making isolation
   the default.

2. **Queue persistence keeps `now` lane volatile.** The plan's edge-case
   list says "`next` and `later` lanes persist; `now` is volatile." I
   enforce this both on read (filter `priority !== 'now'` from restored
   snapshot) and on write (filter before serializing). This means a `now`
   command that was queued during a graceful shutdown is dropped, which
   is the intended trade-off — `now` priority is reserved for messages
   like interrupts and plan-mode confirmations whose context expires
   when the surface is no longer running.

3. **CLI port mirrors contract, not code.** The Rust queue lives in
   `apps/cli/src/message_queue.rs` rather than being imported from a
   shared crate. Reasons: (a) Rust crate boundaries in this repo
   (`crates/*`) are scoped to non-CLI shared logic, (b) the CLI's queue
   integrates with `tokio` async cancellation differently from the
   browser's `AbortSignal`, and (c) the canonical TS implementation can
   evolve without forcing a coupled Rust release. Both implementations
   agree on the same behavioral contract (priority order, FIFO within
   lane, lane cap, atomic compare-and-swap, popAllEditable). The Rust
   port also has its own 11-test suite including the property test.

## 6. What this enables

The queue is the single point of admission for every LLM-bound prompt
across all six surfaces. It gives downstream phases a clean place to
attach:

- **Backpressure** — Sprint B can defer dequeue when the streaming client
  reports rate-limit headers; the lane cap is already there.
- **Replay** — `popAllEditable` restores a queued command into the input
  buffer, so ESC/UP key handling on each surface can let the user edit
  pending sends before they fire.
- **Cancellation** — every existing AbortController in the codebase
  (mobile chatStore L113-462, desktop streaming services, etc.) can now
  be wired into the queue's per-command `signal` slot to give a single,
  uniform cancel surface.
- **Cross-provider continuity** — the queue's `value` field accepts both
  string and `ContentBlock[]`, so multi-block prompts (text + image)
  flow through the same code path as plain text. This is groundwork for
  the "differentiator #3" claim in MEMORY.md.
