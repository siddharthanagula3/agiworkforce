## 2. Schema Diff: Canonical vs Legacy

### 2.1 Timestamp collision — CRITICAL FINDING

Two timestamps have **different SQL content** in canonical vs legacy dirs:

**`20260505000001`:**

- Legacy: `20260505000001_add_api_key_prefix.sql` — adds `key_prefix TEXT` to `api_keys` via `BEGIN/COMMIT` transaction, creates `UNIQUE INDEX`, longer comment block.
- Canonical: `20260505000001_api_keys_key_prefix.sql` — adds `key_prefix text` to `api_keys`, creates non-unique `INDEX` (named `api_keys_key_prefix_idx` vs `idx_api_keys_key_prefix_unique`), shorter comment block.
- **SQL effect difference:** Legacy creates a `UNIQUE` partial index; canonical creates a non-unique partial index. Index names differ. Both add the same column.

**`20260505000002`:**

- Legacy: `20260505000002_replace_authrole_with_role_grant.sql` — bulk rewrites all `auth.role() = 'service_role'` RLS policies to `TO service_role` form via a PL/pgSQL loop.
- Canonical: `20260505000002_fix_github_installations_update_with_check.sql` — targeted fix only for the `github_installations` UPDATE policy WITH CHECK clause.
- **SQL effect difference:** Completely different operations. Legacy touches 20+ policies across all tables; canonical touches one policy.

**Production impact of this collision:** Per the Supabase CLI behavior, migration rows in `supabase.migrations` are keyed by the migration **filename** (which includes the timestamp prefix). If prod DB was applied from the legacy dir first (which it was, per MEMORY.md), it has rows for `20260505000001_add_api_key_prefix.sql` and `20260505000002_replace_authrole_with_role_grant.sql`. When the canonical CLI is run pointing to `supabase/migrations/`, the CLI will see filenames `20260505000001_api_keys_key_prefix.sql` and `20260505000002_fix_github_installations_update_with_check.sql` as **not yet applied** (different filenames) and will attempt to apply them again on top of already-applied content.

**Risk:** Running `supabase db push` from canonical CLI context would attempt to re-apply `20260505000001_api_keys_key_prefix.sql` (create index with different name on column that already exists) and `20260505000002_fix_github_installations_update_with_check.sql` (ALTER POLICY on github_installations — may succeed idempotently, but the bulk policy rewrite from legacy will already have been done). The column add has `IF NOT EXISTS` and the index has `IF NOT EXISTS`, so both would likely be non-destructive. But it would create an extra non-unique index `api_keys_key_prefix_idx` alongside the existing unique index `idx_api_keys_key_prefix_unique`.

**This collision must be resolved before prod push. See Section 6.**
