## Architecture Overview

The feature is split into **two conceptually distinct systems** that share UI real estate:

### 1. Artifact System (primary, fully wired)

The artifact system manages versioned, typed content objects generated by LLM responses. The data flow is:

```
LLM Response
    |
    v
[Chat streaming detects artifact content]
    |
    v
[Frontend calls artifact_create / artifact_create_streaming via invoke()]
    |
    v
[Rust ArtifactStore (in-memory HashMap, parking_lot::RwLock)]
    |
    v
[ArtifactRenderer transforms raw content into type-specific RenderedContent]
    |
    v
[Frontend fetches via artifact_get_rendered]
    |
    v
[ArtifactPanel / ArtifactRendererView renders with type-appropriate viewer]
```

**Key design decisions:**
- **In-memory storage** -- artifacts live in a `HashMap<String, Artifact>` protected by `parking_lot::RwLock`, not persisted to SQLite. The `messages` table has an `artifacts TEXT` column for chat-message-level artifact references.
- **Version history is per-artifact** -- each `Artifact` carries a `Vec<ArtifactVersion>` with content snapshots, hashes, and timestamps. Versions are pruned when they exceed `max_versions` (default 50), keeping the first and most recent entries.
- **Content deduplication** -- `update_content()` computes a `DefaultHasher`-based hash and skips creating a new version if the content hash matches the previous version.
- **Streaming support** -- artifacts can be created in `Streaming` status with empty content, incrementally appended to via `append_streaming`, and finalized into `Complete` status. The frontend polls the rendered artifact at 100ms intervals during streaming.
- **Diff-based updates** -- `artifact_apply_diff` accepts hunks (start_line, end_line, original_content, new_content), validates content matches, applies in reverse line order to preserve offsets, then delegates to the standard `update` path. Frontend has a local fallback via `diffUtils.ts` (LCS-based) if the Rust command is unavailable.

### 2. Canvas System (A2UI protocol, partially wired)

The canvas system provides two capabilities:

**A. Drawing Whiteboard** (`CanvasWorkspace.tsx`) -- a client-side HTML5 `<canvas>` drawing surface with shapes, freehand, text, pan/zoom, undo/redo, and PNG export. Fully self-contained in React state, no Rust backend involvement.

**B. A2UI Protocol** (`features/canvas/a2ui.rs`) -- an Agent-to-UI protocol that allows AGI agents to programmatically compose visual elements (charts, tables, forms, code blocks) on named canvases. Uses `CanvasManager` with event-based updates. **However, the canvas Tauri commands are defined but NOT registered in `lib.rs`** -- the `canvas_*` commands are missing from the `generate_handler!` macro invocation, meaning they are currently dead code from the IPC perspective.

## Artifact Types

Eight artifact types, each with dedicated metadata structs and renderer:

| Type | Rust Enum | Metadata | Renderer | Frontend Viewer |
|------|-----------|----------|----------|-----------------|
| **Code** | `ArtifactType::Code` | `CodeMetadata` (language, file_path, highlight_lines, executable) | `CodeRenderData` | `react-syntax-highlighter` (Prism/oneDark) with line highlighting |
| **Document** | `ArtifactType::Document` | `DocumentMetadata` (format, toc, word_count) | `DocumentRenderData` | `react-markdown` for markdown format, plain `<div>` for others |
| **Spreadsheet** | `ArtifactType::Spreadsheet` | `SpreadsheetMetadata` (columns, row_count, column_types, formulas) | `SpreadsheetRenderData` | HTML `<table>` with headers and row numbering |
| **Diagram** | `ArtifactType::Diagram` | `DiagramMetadata` (diagram_type, theme) | `DiagramRenderData` | Dynamic import of `mermaid` library, SVG sanitized via `sanitizeSvg()` |
| **Web** | `ArtifactType::Web` | `WebMetadata` (enable_scripts, external_resources, viewport) | `WebRenderData` | Sandboxed `<iframe>` with CSP meta tag, `sandbox="allow-scripts allow-modals"` |
| **Chart** | `ArtifactType::Chart` | `ChartMetadata` (chart_type, x_label, y_label, show_legend) | `ChartRenderData` | `recharts` (BarChart, LineChart, PieChart) in `<ResponsiveContainer>` |
| **Presentation** | `ArtifactType::Presentation` | N/A (uses generic) | `PresentationRenderData` | Slide navigator with prev/next, markdown rendering per slide |
| **Image** | `ArtifactType::Image` | N/A (uses generic) | `ImageRenderData` | `<img>` tag with base64 or URL source |

**Type detection:** Artifact types are specified explicitly at creation time (not auto-detected from content). The Rust `ArtifactStore::create` assigns default metadata based on type if none is provided.

**Default file extensions and MIME types** are defined on `ArtifactType` -- e.g., `Code -> "txt"/"text/plain"`, `Document -> "md"/"text/markdown"`, `Web -> "html"/"text/html"`.

## Canvas Editor

### Artifact Code Editor (`CanvasPanel.tsx` + `CodeEditor.tsx`)

The Canvas code editor provides a lightweight editing experience:

- **Layout:** Header (title input, type badge, language badge) -> Tabs (Code | Preview) -> Content area -> Toolbar (Run, Copy, Export, Delete)
- **Editor:** Styled `<textarea>` with line numbers, tab-inserts-2-spaces, language label, copy button
- **Preview:** Type-dependent -- sandboxed iframe for HTML, regex-based markdown renderer for markdown, terminal-style output for executed code, formatted text for documents
- **Code execution:** Supports Python, JavaScript, and Bash via `execute_code` or `terminal_execute` Tauri commands with graceful fallback
- **Bug fix flow:** Error banner with "Fix Bug" button that can relay the error message back to the chat for AI-assisted debugging

### Artifact Panel (`ArtifactPanel.tsx`)

The full-featured artifact panel used alongside chat:

- **Tabbed interface** -- each artifact in the conversation gets its own tab
- **Toolbar:** Status badge, version indicator (`v{N}`), relative timestamp, History button, Edit button, Copy, Download, More menu (Share, Pin, Archive, Delete)
- **Inline editing** via `InlineArtifactEditor` -- computes LCS-based line diff on save, applies via `applyDiffToArtifact`
- **Version history** via `VersionHistoryDialog` -- lists all versions with timestamps, sizes, hashes; click to select, Restore button creates a new version with old content
- **Resizable** -- drag handle on the left edge, 320px min to 800px max
- **Expandable** -- maximize button goes full-screen overlay
- **Sharing** via `ShareArtifactDialog` -- generates shareable URL with two-tier strategy (base64 for <4KB or no auth, Supabase for larger content)

### Drawing Canvas (`CanvasWorkspace.tsx`)

A standalone freeform drawing workspace:

- **Tools:** Select (V), Pan (H), Rectangle (R), Circle (C), Line (L), Text (T), Freehand (P)
- **Features:** Stroke/fill color pickers with 10 preset colors, stroke width slider (1-20px), undo/redo with full history, zoom (scroll wheel + buttons, 10%-500%), grid background, element selection with dashed blue outline
- **Export:** PNG download via `canvas.toDataURL()`
- **Input:** Keyboard shortcuts for all tools, Delete/Backspace removes selected, Cmd+Z/Y for undo/redo
- **Rendering:** HTML5 Canvas 2D context with transform (pan + zoom), grid lines, element drawing loop

## Store Schemas

### `artifactStore.ts` (Zustand + Persist)

```typescript
interface ArtifactStoreState {
  // Data
  artifacts: Map<string, Artifact>;      // Client-side cache (NOT persisted -- Map serializes as {})
  summaries: ArtifactSummary[];           // List view cache

  // Selection
  activeArtifactId: string | null;        // Currently viewed artifact
  selectedVersion: number | null;         // Selected version in history view

  // Panel
  panelOpen: boolean;                     // Side panel visibility
  panelWidth: number;                     // Panel width in px (default 480, persisted)

  // Loading
  isLoading: boolean;                     // General loading flag
  isStreaming: string | null;             // ID of currently streaming artifact

  // 30+ action methods (see IPC table above for the Tauri-backed ones)

```

**Persistence:** Only `panelWidth` is persisted to localStorage (`artifact-store` key, version 1). The `artifacts` Map is intentionally excluded because `JSON.stringify` serializes Maps as `{}`.


## Key Patterns

### Artifact Detection in Chat Responses

Artifacts are attached to chat messages via the `artifacts` field on message objects (stored as `artifacts TEXT` in the SQLite `messages` table). The `ArtifactsView` component iterates over `messagesByConversation` and collects all `msg.artifacts` into a flat searchable list. The `ArtifactRenderer` in `UnifiedAgenticChat/` handles inline display within the chat stream.

### Version Management

- **Creation:** `Artifact::new()` creates version 1 automatically
- **Streaming:** `new_streaming()` creates version 0, `finalize_streaming()` promotes to version 1
- **Updates:** `update_content()` hashes content, skips if unchanged, otherwise increments version number
- **Rollback:** `rollback_to_version()` copies old content into a NEW version (non-destructive -- version history is append-only)
- **Pruning:** When `versions.len() > max_versions` (50), keeps the first version and the most recent `max_versions - 1` versions
- **Diff viewing:** `get_version_diff()` returns raw content of both versions for frontend diffing

### Diff-Based Updates

Two complementary diff systems:

1. **Rust-side** (`artifact_apply_diff`): Accepts `ArtifactDiffHunk` structs with `start_line`/`end_line`/`original_content`/`new_content`. Validates content matches (trimmed comparison), applies hunks in reverse line order, delegates to `update()` for versioning.

2. **Frontend-side** (`diffUtils.ts`): LCS-based `computeLineDiff()` produces hunks from original vs modified text. `applyDiff()` reconstructs the modified text from original + hunks. Used by `InlineArtifactEditor` to compute minimal diffs on save, and as a fallback if the Rust `artifact_apply_diff` command is unavailable.

### Sharing

Two-tier strategy in `artifactSharing.ts`:
- **Base64** (content < 4KB or no auth): Encodes artifact as JSON, base64-encodes, embeds in URL query parameter (`?data=<encoded>`). No server storage needed.
- **Supabase** (content >= 4KB, user authenticated): Inserts into `shared_artifacts` table, returns UUID-based URL (`/shared/{uuid}`). Falls back to base64 on any failure. Supports view count tracking and expiry.

### Rendering Pipeline

The Rust `ArtifactRenderer::render()` transforms raw `Artifact` content into `RenderedContent` (tagged union), which includes:
- Type-specific structured data (e.g., `CodeRenderData` with parsed language, line count, file extension)
- Available actions list (Copy, Download, Edit, Delete, Pin + type-specific: Run, ApplyToFile, ExportPdf, ExportSvg, etc.)
- Version info (current version, total count, timestamps)

The frontend `ArtifactRendererView` dispatches on `rendered_content.type` to the appropriate sub-renderer.

### A2UI Protocol (Agent-to-UI)

The A2UI protocol allows AGI agents to compose visual elements programmatically:
- **Commands:** CreateCanvas, AddText, AddImage, AddChart, AddTable, AddForm, AddMarkdown, AddCode, UpdateElement, RemoveElement, ClearCanvas, ShowNotification
- **Auto-layout:** Default positions (20,20) and sizes (400x200) when coordinates are not specified
- **Active canvas:** Commands with `canvas_id: None` target the active canvas
- **Events:** `CanvasManager` emits `CanvasEvent`s (CanvasCreated, ElementAdded, ElementUpdated, etc.) via `tokio::mpsc` channel for frontend reactivity
- **Element types:** Text, Image, Chart (7 chart types), Table, Form (11 input types), Markdown, Code, Button (5 variants), Container (4 layouts), Divider, Progress

## Known Issues / Tech Debt

1. **Canvas commands not registered** -- All 13 `canvas_*` Tauri commands in `sys/commands/canvas.rs` are defined and compile, but are NOT registered in the `generate_handler!` macro in `lib.rs`. The `CanvasStateManager` IS initialized via `app.manage()`, but the commands cannot be invoked from the frontend. The A2UI protocol is effectively dead code from an IPC perspective.

2. **artifact_export_all and artifact_import_all not registered** -- These two backup/restore commands are defined in `sys/commands/artifacts.rs` but not listed in `lib.rs`'s command registration. Similarly `artifact_clear_all`.

3. **In-memory artifact storage** -- `ArtifactStore` uses an in-memory `HashMap`, not SQLite. Artifacts are lost on app restart. The `messages.artifacts` column stores references but the actual artifact content/versions are transient.

4. **Two parallel artifact systems** -- `canvasStore.ts` and `artifactStore.ts` manage overlapping concepts (artifacts with types, content, editing) but are not connected. `canvasStore` is frontend-only with code execution; `artifactStore` is Rust-backed with versioning. A unified system would reduce complexity.

5. **No artifact persistence to disk** -- Despite having export/import commands (unregistered), there is no automatic persistence of artifact content or version history to SQLite or the filesystem.

6. **ArtifactRenderer in UnifiedAgenticChat** (~1515 lines) -- This standalone renderer duplicates rendering logic from `ArtifactRendererView`. It uses `codeStore` and `chat` types separately from the artifact system. This should be consolidated.

7. **Mermaid diagram security** -- The `DiagramRenderer` uses `sanitizeSvg()` (added as AUDIT-NEW-001 fix) but the Mermaid library itself is dynamically imported with `securityLevel: 'strict'`. The SVG is still rendered via `dangerouslySetInnerHTML`.

8. **Web artifact sandbox** -- HTML previews use `sandbox="allow-scripts allow-modals"` without `allow-same-origin`, which is intentionally restrictive. CSP meta tags are injected but external resource loading (`WebMetadata.external_resources`) is not wired to the CSP policy.

9. **Streaming poll interval** -- The `ArtifactPanel` polls `getRenderedArtifact` every 100ms during streaming. This is functional but could be replaced with Tauri event-based updates for better efficiency.

10. **Chart renderer type coverage** -- Only bar, line, and pie charts are rendered via recharts. The remaining types supported by `ChartMetadata` (scatter, area, radar, doughnut) show "Unsupported chart type" in the frontend.
